# Article Evidence Pack (Script/Result/Data Audit)

Created: 2026-03-20  
Purpose: gather grounded evidence for writing an accurate project article (without drafting the article itself).

## 1) Scope and Ground Rules

- Final accepted irradiance -> generation model: `scripts/python_physics/physics_model.py`.
- Other irradiance -> generation models in repo are exploratory/approximation paths and were not selected as final reference.
- This evidence pack focuses on:
  - what was implemented,
  - what was tested/discarded,
  - what outputs prove,
  - what data files were used.

## 2) Final Model Chosen

## 2.1 Chosen path
- `scripts/python_physics/physics_model.py`

## 2.2 Why this path was kept
- Most complete physics chain in repo:
  - Perez POA via pvlib
  - AOI + IAM interpolation
  - SAPM thermal model
  - DC loss chain in PVsyst order
  - load-dependent inverter curve support
  - plant-level clipping and AC wiring loss
- Config-driven (plant parameters in `config/plant_config.json`) rather than hardcoded workflow-only constants.

## 2.3 Code evidence highlights
- Inverter mode switch in final model:
  - `USE_INVERTER_CURVE = config["inverter"]["use_efficiency_curve"]`
  - curve interpolation in `inverter_efficiency(...)`
- Plant-level clipping:
  - `plant_ac = plant_ac.clip(upper=INV_AC_RATING_KW)`
- Daily full-physics path and simplified daily path both exist:
  - `generate_synthetic_hourly_from_daily(...)`
  - `calculate_daily_energy_simple(...)`

## 3) Trial-and-Error Models (Not Final)

| Candidate | Files | What it tried | Key result | Why not final |
|---|---|---|---|---|
| JS Physics approximation | `scripts/js_physics/physics_model.js`, calibration scripts | ThingsBoard-friendly fast physics approximation with calibrated Perez/SAPM terms | Good parity to Python labels on calibration/validation sets | Still simplified runtime model (flat inverter efficiency, simplified geometry/model choices) vs final Python reference chain |
| JS Surrogate regression | `scripts/js_surrogate/surrogate_model.js`, `fit_surrogate.py` | Ultra-fast linear/quadratic feature model of power | Very low hourly fidelity despite low net energy error | R2 about 0.42 on test set; unsuitable for hourly anomaly/performance monitoring |
| Simplified daily energy model | `physics_model.py --simplified` | Fast daily estimate from daily totals | Operationally fast | Lower-fidelity daily approximation; bypasses hourly physics dynamics |
| Inline helper physics variants | `daily_predictor.py`, `prepare_training_data.py`, `run_benchmark.py`, `evaluate_models.py` temp JS | Convenience/benchmark/training execution paths | Useful for pipelines and comparisons | Not identical to final `physics_model.py` reference path (notably flat inverter efficiency in helpers) |

## 4) Output Evidence (Important Files)

## 4.1 JS physics calibration quality
Source file: `output/js_physics_calibration.json`
- Baseline RMSE: 2.2449 kW
- Final RMSE: 1.4340 kW
- MAE: 0.6723 kW
- R2: 0.99233
- Energy error: 1.491%
- Improvement: 36.12%
- Optimization: 112 iterations, 13,677 evaluations

Interpretation:
- Calibration succeeded technically for JS approximation against Python-generated labels.

## 4.2 JS vs Python validation on large synthetic set
Source file: `output/js_validation_results.json`
- Samples: 2,500,000
- RMSE: 1.4577 kW (2.65% of 55 kW)
- MAE: 0.6763 kW
- R2: 0.99214
- MAPE: 7.76%
- Energy error: 1.456%

Interpretation:
- Strong approximation quality in synthetic validation domain.

## 4.3 Held-out 2024 test metrics (JS models vs Python labels)
Source file: `output/test_evaluation_metrics.json`
- Test samples: 1,357
- Reported test days: 131 (UTC-normalized)

JS Physics:
- MAE: 1.3387 kW
- RMSE: 3.2810 kW
- R2: 0.9685
- MAPE: 20.09%
- Energy error: +5.47%

JS Surrogate:
- MAE: 9.4375 kW
- RMSE: 14.0893 kW
- R2: 0.4200
- MAPE: 129.70%
- Energy error: +1.50%

Interpretation:
- JS surrogate kept only as an experiment/edge case for coarse totals.
- It is not acceptable for hourly expected-generation baselines.

Important provenance note:
- These metrics are generated by `scripts/shared/evaluate_models.py`, which executes an inline JS physics path (temporary script), not a direct runtime invocation of `scripts/js_physics/physics_model.js` as deployed.

## 4.4 Benchmark vs Meteocontrol actuals
Sources:
- `output/benchmark_metrics.json`
- `output/benchmark_results.csv`
- `output/benchmark_report.md`

`benchmark_metrics.json` reports (5-day window label):
- Python Physics MAE: 154.13 kWh/day
- JS Physics MAE: 168.98 kWh/day
- JS Surrogate MAE: 151.29 kWh/day
- Very large total_error_pct values (+239% to +268%)

Critical caveat from `benchmark_results.csv`:
- CSV contains only 2 overlap rows with non-null actuals (Dec 11 and Dec 12).
- Overlap-only total errors are much lower than JSON total_error_pct:
  - Python: +15.81%
  - JS Physics: +27.03%
  - JS Surrogate: +18.29%

Interpretation:
- Window completeness and alignment materially affect headline errors.
- Any article claim should explicitly state overlap size and missing-actual handling.

## 4.5 Source comparison against actuals
Source: `output/accuracy_report.txt`

Daily (10-day matched window):
- NASA POWER: MAE 96.2 kWh, R2 -0.68
- Solcast: MAE 97.4 kWh, R2 -0.71
- Open-Meteo: MAE 111.4 kWh, R2 -1.55
- ERA5: MAE 96.8 kWh, R2 -0.70

Interpretation:

- Negative R2 here indicates poor fit in that window, not necessarily a broken physics chain.

## 6) What Is Safe to Claim in a Future Article

High-confidence claims:
- Final reference model is Python pvlib chain in `physics_model.py`.
- JS physics model was calibrated successfully to match Python labels on synthetic/validation datasets.
- JS surrogate is fast but poor on hourly fidelity (R2 ~0.42).
- Weather-input quality is a dominant error source on cloudy windows.

Claims that need explicit caveats when used:
- Any benchmark “total error %” from `benchmark_metrics.json` (state overlap incompleteness).
- Any JS-vs-Python test metric from `evaluate_models.py` (state inline evaluator path).

## 7) Repository Hygiene Findings Relevant to Writing

- `docs/project_plan.md` is now rewritten to align with final model methodology.

## 8) Suggested Citation Pattern for Internal Consistency

When citing a metric in writing, include:
1. metric value,
2. generating file,
3. generating script/path,
4. dataset/window description.

Example format:
- "RMSE 1.4577 kW (output/js_validation_results.json, generated by scripts/js_physics/validate_calibration.py on 2.5M synthetic samples)."
